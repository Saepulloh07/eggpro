================================================================================
SIMULASI EDGE CASES & HASIL PENGUJIAN - POULTRYMIND ERP
================================================================================

1. Double Submit Transaksi
--------------------------------------------------------------------------------
SkenARIO: User menekan tombol "Simpan" berkali-kali pada modal Biaya.
HASIL: WARNING. Meskipun backend menggunakan transaksi, tidak ada pengecekan idempotensi (misal: unique request ID). Jika request sampai ke backend 2 kali, maka akan tercipta 2 transaksi dan 2 jurnal.
REKOMENDASI: Implementasikan disable button saat isSaving = true.

2. Jaringan Terputus saat Transaksi
--------------------------------------------------------------------------------
SKENARIO: Koneksi mati tepat setelah tombol simpan ditekan, sebelum response diterima.
HASIL: PASS. Aplikasi menggunakan syncRecord yang menyimpan data di IndexedDB (localforage) sebagai fallback, namun untuk transaksi finansial yang menggunakan API (POST), data mungkin tidak tersimpan di backend tapi tersimpan di state lokal (inkonsistensi).
REKOMENDASI: Gunakan antrian (Queue) untuk transaksi API yang gagal.

3. Edit Transaksi di Periode yang Sudah Closing
--------------------------------------------------------------------------------
SKENARIO: User mencoba mengubah data transaksi bulan lalu yang sudah di-close.
HASIL: PASS. Fungsi checkLockAndSwal di GlobalContext.tsx memblokir aksi ini dengan baik berdasarkan farmSettings.lastClosingDate.

4. Stok Menjadi Negatif
--------------------------------------------------------------------------------
SKENARIO: User menginput pemakaian pakan 100kg padahal stok hanya 50kg.
HASIL: PASS. Backend (transaction.ts:46) melakukan pengecekan `quantity >= ?` dalam query UPDATE. Frontend (GlobalContext.tsx:254) juga memblokir stok negatif.

5. Penghapusan Master Data (Account)
--------------------------------------------------------------------------------
SKENARIO: Menghapus akun "Kas Utama" yang sudah memiliki ribuan baris jurnal.
HASIL: FAIL. Database akan melempar error Foreign Key Constraint. UI mungkin akan crash jika tidak menangani error ini dengan anggun.
REKOMENDASI: Tambahkan flag `isSystem` untuk akun-akun dasar agar tidak bisa dihapus. (Sudah ada di types.ts, pastikan UI menghormatinya).

6. Multi-User Update Populasi
--------------------------------------------------------------------------------
SKENARIO: Dua pekerja menginput mortalitas secara bersamaan untuk kandang yang sama.
HASIL: PASS. Backend menggunakan `UPDATE flock_batches SET currentCount = currentCount - ?` yang bersifat atomik di level database.

7. Penjualan Tanpa Harga (Rp 0)
--------------------------------------------------------------------------------
SKENARIO: User menginput penjualan telur tapi lupa mengisi harga.
HASIL: PASS. Sistem mencatat sebagai INCOME Rp 0. Jika ditandai "Free", maka masuk kategori "Free Goods".

================================================================================
RINGKASAN:
Sistem memiliki ketahanan yang baik terhadap integritas stok dan periode closing. Kelemahan utama ada pada potensi double-submit dan penanganan error FK pada penghapusan master data.
================================================================================
